# 02. 設計思想・基本原則

## 1. FrontierAtlas の設計理念

1. **研究成果をそのまま実運用へ移行できること**
2. **変更しやすく、壊れにくい構造であること**
3. **プラットフォームに依存しないこと**
4. **各レイヤが独立して進化できること**
5. **データとロジックを明確に分離すること**

これらをすべての設計判断の基準とする。

## 2. 基本設計原則

1. **単一責務（Single Responsibility）**: 各モジュールは一つの責務のみを持つ
2. **疎結合（Loose Coupling）**: モジュール間の依存を最小化
3. **高凝集（High Cohesion）**: 関連する責務を一箇所に集約
4. **インターフェース指向**: 実装ではなくインターフェースに依存
5. **レイヤ分離**: 上位レイヤは下位レイヤの実装詳細を意識しない
6. **イベント駆動**: ポーリングせず、必要な変化のみ通知
7. **データ駆動**: ロジックではなくデータ変更で振る舞いを変える

実装時は、短期的な実装容易性よりも長期的な保守性を優先する。

## 3. レイヤ分離（3層構造）

```text
Application Layer（React Native・MapLibre）
──────────────
Engine Layer（Android Position Engine）
──────────────
Data Layer（Map Assets・Repository）
```

各レイヤは公開 API を通じてのみ連携する。詳細は `03_system-architecture.md` 参照。

## 4. 責務分離マトリクス

| モジュール | 責務 |
|----------|------|
| MapLibre | 地図描画 |
| React Native | UI・状態管理 |
| Position Engine | 測位 |
| Repository | データ取得 |
| Map Assets | データ管理・配布 |

複数の責務を一つのモジュールへ集約しない。一つの責務を複数モジュールへ分散させない。

## 5. UI とロジックの分離

UI は表示のみ担当。測位計算・Graph 計算・Fingerprint 判定・センサー制御を禁止。React Native は Position Engine の出力のみを利用する。

## 6. ネイティブ処理の原則

Android 固有機能（Sensor・Wi-Fi・BLE・Permissions・Foreground Service・Coroutine）はすべて Kotlin 側へ集約する。React Native では Android API を直接扱わない。

## 7. API 設計原則

公開 API はシンプルに保つ（`initialize()` / `start()` / `pause()` / `resume()` / `stop()`）。内部構造を公開しない。

## 8. パフォーマンス・非同期処理・キャッシュ戦略

- **パフォーマンス**: 不要なイベント送信・全 Layer 再描画・毎秒ファイル読み込み・同一データ再生成を禁止。差分更新を基本とする。
- **非同期処理**: Assets 読込・Fingerprint 読込・Graph 読込・Route 生成はすべて非同期化。UI スレッドをブロックしない。
- **キャッシュ戦略**: Building・Floor・Graph・Fingerprint をキャッシュ。毎回ファイルアクセスしない。

## 9. テスト容易性

すべて Interface を持ち、DI（依存性注入）により実装を差し替え可能な構造とする。例：`Repository → Interface → Mock Repository`

## 10. エラーハンドリング

例外を UI まで伝播させない。`Engine → Error Event → Bridge → UI` の流れで、UI はイベントとして受け取る。

## 11. ログ設計

ログレベルを統一する：

| レベル | 用途 |
|-------|------|
| **DEBUG** | 開発時の詳細情報 |
| **INFO** | 通常動作の記録 |
| **WARN** | 回復可能な問題 |
| **ERROR** | 要対応のエラー |
| **FATAL** | 継続不能な致命的エラー |

研究用ログと運用ログは分離する。研究用ログはセンサー生データ・中間推定値・アルゴリズム比較用メトリクスを含み、運用ログはエラー・警告・イベント通知履歴に限定する。

## 12. 開発プロセス方針

開発は以下の順序を基本とする：

```text
設計 → インターフェース定義 → 実装 → テスト → リリース
```

設計を先行させ、実装は設計に従う。特に以下を遵守する：

- 新機能追加時は必ずインターフェース定義から始める
- 実装前に他レイヤへの影響を評価する
- 設計書の更新を実装と同時に行う
- テストは単体→統合→E2E の順で実施する

## 13. 命名規則

命名は責務を表現する。良い例：`PositionRepository` / `RouteService` / `FloorSelector`。悪い例：`Manager2` / `Helper` / `Util` / `DataClass2`。曖昧な名前を避け、ディレクトリは技術ではなく責務で分割する。

## 14. 設計判断基準

設計で迷った場合は以下の順序で判断する：

1. 責務分離を守れるか
2. レイヤ境界を越えていないか
3. 公開 API を変更せず実現できるか
4. パフォーマンスへ悪影響がないか
5. テストしやすいか
6. 実装が理解しやすいか

## 15. AI 活用を前提とした設計

- ディレクトリ構造を明確にする
- 責務を明確にする
- インターフェースを維持する
- コメントとドキュメントを整備する

AI が安全にコードを理解・変更できる環境を維持することも設計品質の一部と考える。
